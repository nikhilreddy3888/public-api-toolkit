The problem

AI is solving easy problems the hard way.

Every time an AI assistant browses for weather, hallucinates exchange rates, or scrapes news HTML for a one-line factual answer — it burns tokens and time that a structured API call would solve in 200ms.

What agents do today (expensive)

  • Search the web for BTC price — burns 800+ tokens parsing finance pages
  • Reason about FX rates from stale training data — answer might be months old
  • Scrape weather.com HTML for tomorrow's forecast — brittle, slow, wasteful
  • Fetch full Wikipedia pages just to summarize one paragraph
  • Install a different MCP server for every single capability

What they should do (instant)

  • Call public_api_crypto_data — structured JSON in <300ms
  • Call public_api_currency_exchange — live rates, zero hallucination
  • Call public_api_weather — clean forecast, 0 HTML parsing
  • Call public_api_wikipedia — structured summary, minimal tokens
  • One npx -y public-api-toolkit covers all 43 tools

Less tokens. Less compute. More signal.

FAQ

Common questions.

Do I need API keys for all 43 tools? +
No. Most tools work without any API keys — they use free-tier public APIs that don't require authentication. Optional premium providers (Finnhub, Alpha Vantage, TMDB, etc.) are unlocked via environment variables like PUBLIC_APIS_FINNHUB, but the core toolkit works out of the box with zero configuration.
How is this different from just letting the agent use web search? +
Web search returns noisy HTML that costs 500–2000+ tokens per lookup. Structured API calls return clean JSON in <300ms using 50–100 tokens. For deterministic factual questions (what's BTC at, is today a holiday, weather in Seoul), structured lookups are faster, cheaper, and more accurate.
Isn't this just the public-apis list wrapped in MCP? +
It's inspired by the same ecosystem but fundamentally different in purpose. public-apis is a curated list of APIs — a reference document. Public API Toolkit is a running MCP server that makes those APIs callable by AI agents as structured tools. It handles the HTTP layer, normalizes responses, manages timeouts and fallbacks, and exposes everything through a single consistent tool surface. No scraping, no glue code, no per-API setup. (Huge thanks to the public-apis project for curating the ecosystem that made this possible.)
What if an upstream API goes down? +
The toolkit has 15-second timeouts and graceful fallback behavior for degraded providers. Some tool groups include fallback behavior, so an upstream problem does not always mean a dead-end request.
Can I contribute new APIs or fix broken ones? +
Yes — contributions are welcome. CONTRIBUTING.md has everything you need. Adding a tool takes about 20–30 minutes following a consistent pattern. This project exists because contributors keep adding safe, useful, free APIs that reduce the need for wasteful web scraping.
